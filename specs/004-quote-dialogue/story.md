现在，spec 的澄清是通过 github pr来完成的。这要求用户熟悉 PR 用法，并且要新建 spec 分支，申请 pr权限，效率也更低。

现在有一个更快的方法，就是使用 markdown 的 quote语法来完成需求澄清。它的语法如下：

```md
# 原文段落

这里是原文的第 1 行内容。例如：“这个算法的时间复杂度是 O(n^2)。”

> **Aaron：** 我认为这里应该是 O(n log n)，而不是 O(n^2)。请核实。
>> **Sage：** 你说得对。我重新检查后，发现使用了归并排序，时间复杂度为 O(n log n)。已修正。

>>> **Aaron：** 那空间复杂度呢？归并排序需要额外 O(n) 空间，这里标注了吗？
>>>> **Sage：** 是的，原段落漏掉了空间复杂度说明。我补充为“空间复杂度 O(n)”。 当 AI 将 story 记录下来之后，对需要澄清的问题，sage 先通过 > **sage**: 来进行提问，用户(假设是 Aaron)通过>> **Aaron**: 来回复。通过'>'的个数来进行缩进。用户也可以直接对文档进行提问，由 Agent来回答
```

当sage 把用户 story 转换为 spec时，就可以对它还不清楚的问题，通过quote语法，要求用户澄清；用户在 IDE 中打开该文件，通过 quote 语法（多一层缩进）来回复AI；此外，他还可以对任意段落进行提问，由Agent回答。
