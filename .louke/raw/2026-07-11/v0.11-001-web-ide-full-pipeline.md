## Devon 不可用根因 (接手人必读)

**症状:** Maestro 用 `task subagent_type=devon` spawn 子 session,4 次任务(包含 1 个极简 "写 1 个 hello 文件" 探针)全部返回空 result,工作区无任何产物。

**日志证据** (从 `~/.local/share/opencode/log/opencode.log`):
```
message=created id=ses_0af855289ffe3rPv6sE22fhtdf agent=devon model=undefined
message=stream providerID=volcengine-plan modelID=kimi-k2.7-code
message=loop step=0
message=loop step=1
message="exiting loop"  ← 1 步后立即退出
```

**对照 (general 探针同时间):**
```
message=stream providerID=m3-local modelID=minimax-m3
message=loop step=0 → step=1 → step=2
message="exiting loop"  ← 2 步,每次 stream 有响应
```

**根因:** `.opencode/agents/devon.md` 配 `model: volcengine-plan/kimi-k2.7-code`。该模型在本环境:
- step 0 启动 stream 后 5 秒内没产生任何 token 输出
- session 直接 "exiting loop" 而没有 "stream completed" 或 token 计数
- general agent (m3-local/minimax-m3) 同样探针正常

可能原因:
- kimi API 配额耗尽 (minimax Token Plan 已用尽, 见 `AI_APICallError: 已达到 Token Plan 用量上限`)
- kimi 模型对短 prompt + tool call 场景下返回空
- devon.md system prompt + kimi 模型组合有兼容问题

**修法 (按优先级):**

1. **首选:** 把 devon.md 的 model 改成本地 fallback。
   编辑 `.opencode/agents/devon.md`:
   ```yaml
   model: m3-local/minimax-m3  # 或 volcengine-plan/deepseek-v4-flash
   ```
   deepseek-v4-flash 在 keeper/scout/warden 已被使用, 稳定。
   minimax-m3 是本地模型, 无外部依赖。

2. **次选:** 修 kimi API 配额或换 endpoint。日志中 minimax plan 提示 `已达到 Token Plan 用量上限`,需要充 Token Plan 或切 endpoint。

3. **保守:** 维持 Louke 纪律, 不改 agent 配置, 把 M-DEV 留给接手人手动执行 (用 IDE 内 chat + R-G-R)。
