## 0. 原始输入

> 为 v0.14.1 立项一个 Story：在 v0.14.0 已锁定并正在发布的基础上，新增 **全局 Story Registry** 和 **Requirement Registry**，并在必要场景下引入 **跨 Spec 的限定 requirement 引用**（qualified reference）。
>
> Human 已确认的关键决策：
> 1. **索引从部署开始构建，仅覆盖新的 Specs**；**不回填**历史 Story / Spec。
> 2. **疑似重复的 Story 与疑似重复/冲突的 requirement 必须阻断流程推进**；Human 可显式豁免/放行。
> 3. **Wiki 的生成/消费在本版本（v0.14.1）显式排除**，不属于本期范围。
> 4. v0.14.0 当前发布工作单独推进，不受本 Story 影响。
>
> Semantic 边界（必须保留）：
> - Scribe 只做 **Story Registry 检查并向 Human 上报疑似重复候选**，**不裁决**重复状态，**不在起草阶段更新 Registry**。
> - Sage 只做 **Requirement Registry 检查并把 superseded / duplicate / conflict 结果纳入独立 review**，**不静默退役或改写既有 requirement**。
> - Runtime 在 Human 决策与 requirement **最终化/锁定后**，**原子或带审计证据地更新 Registry**。Registry 是已批准 identity / relationship 的索引，不是 Story/Spec 内容的第二事实源。
> - Registry **从部署时为空/新建**，**只覆盖新的 Specs**；历史资产保持可作为普通文件检索，**不被注册/回填**。
> - "疑似重复/冲突必须阻断；Human 可豁免" 要求 **显式 disposition 与可追溯**，不是自动语义拒绝。
> - 跨 Spec 限定引用 **仅在存在跨 Spec 歧义时使用 Spec 命名空间**（如 `FR-2200@v0.14-002`）；**本 Spec 内引用保持短形式**。不要过早过度规范 parser 语法。
> - **不**把 Wiki 生成/消费纳入 v0.14.1 范围。

## 1. 用户意图

- **主要用户**：使用 Louke 推进产品发布的产品负责人兼维护者（与 v0.14 同构），以及由 Runtime 调度的语义协作者 Scribe / Sage / Lex / Prism。
- **当前处境**：v0.14.0 已锁定并正在发布，product 的 Story/Spec 数量随时间累积。Scribe 在新 Story 起草阶段只能依赖文本比对和作者记忆判断"是否已有相同主题"；Sage 在新 Spec 起草阶段只能依赖 issue/AC 文本判断"是否已有相同 requirement"。跨 release 复用时（例如 v0.14-002 的 `FR-2200` Archer 规范性合同在 v0.14.1 起草阶段被再次提出），既无机器可查的索引，也无法用规范语法在不同 Spec 间稳定引用对方已批准的 requirement。
- **目标结果**：
  1. Scribe 在 Story 起草阶段可以查询一个 **只读 Story Registry**（覆盖新 Specs）并向 Human 报告 **疑似重复 Story 候选**；Human 显式处置后才放行后续阶段。
  2. Sage 在 Spec 起草前与起草过程中可以查询一个 **只读 Requirement Registry**（覆盖新 Specs），将 superseded / duplicate / conflict 候选作为 **review context** 上报；Lex 在 M-SPEC review 阶段**独立重新查询**同一 Registry 并把同一类候选纳入独立 review，由 Human 显式处置后才放行锁定。
  3. 在跨 Spec 引用场景下，**只有当局部 ID 存在歧义**时才使用 `FR-NNNN@<spec-id>` 限定语法；同 Spec 内的引用保持短形式。
  4. Wiki 生成/消费在本版本**显式不在范围内**；Registry 不作为 Wiki 的输入或输出。
- **成功信号**：
  - 自 v0.14.1 cutover identity（见 §2.1 第 1 步）起，每一个进入 M-SPEC 的新 Spec，其 approved FR/NFR 都可在 Requirement Registry 中查到，且可被任何后续 Spec 用短形式或限定形式稳定引用。
  - 起草阶段与 review 阶段分别产生显式的 disposition record（actor / reason / target / revision / evidence / disposition），候选处置不依赖作者记忆。
  - 跨 release / 跨 Spec 引用在 Lex 解析层可识别 namespace，可与短形式共存而不产生歧义。

## 2. 核心操作路径

### 2.1 主路径

- **起点上下文**：
  - 主路径 A（Story 注册）：用户已经在 Web Chat 中提交新 Story 设想，Runtime 已创建 run 并 dispatch Scribe。M-STORY 的"调查 + 撰写"阶段开始。
  - 主路径 B（Requirement 注册）：Sage 在 M-SPEC 起草中；Lex 在 M-SPEC review 阶段对 Sage 修订后的 spec.md 进行独立 review。
- **入口/触发**：
  - A 路径由 Scribe 的 Story discovery 标准流程触发，**不新增用户入口**。
  - B 路径由 Sage 的 Spec drafting 与 Lex 的 review 流程触发，**不新增用户入口**。
  - Registry 查询与更新是 program/agent boundary 的内部接口，**不暴露给用户界面**作为独立按钮；用户只在原有 gate（Go / Park / No-Go / requirements approval / M-LOCK）交互中看到 disposition 与处置记录。

1. **Registry cutover identity 与初始化**：v0.14.1 部署阶段由 Runtime 创建并持久化一个可被引用的 **cutover identity**，至少包含：
   - `registry_baseline` digest：由 Runtime 在 v0.14.1 安装/初始化时生成的、绑定到当前部署的 `registry_baseline` 摘要（来源：Runtime 创建的 deployment manifest / activation record），用于判定一个 Spec 是否进入新 Registry 的时间分界；
   - 配套的 registry activation record（actor、时间戳、基础 manifest digest、目标项目/workspace identity）。

   初始化时两个 Registry 均为空，**只承认 Runtime identity / creation / admission record 严格晚于 `registry_baseline` digest 的 Spec**，且该 Spec 必须通过 v0.14.1 新 admission 合同（co-designed in M-SPEC）才被索引。历史资产（包括 v0.14.0 及更早版本已批准 Story / Spec / FR / NFR，以及 cutover 当日尚未以新 admission 合同重新进入流程的草稿）保持可作为普通文件检索，但**不被注册、不被回填**。即便这些历史资产在 cutover 之后被外部修改或被引入到新 run 中，只要它们没有以新 admission 合同重新创建为 post-cutover Spec，就不会被 Registry 索引。

2. **Story 起草前查询（A 路径）**：Scribe 在 draft `story.md` 之前，program boundary 提供 `story_registry.query({scope, topic, stage}) -> candidates[]`；Scribe 据此生成 **疑似重复 Story 候选清单**（不含自动决策），并作为 story.md 的输入上下文写入。

3. **Story 候选上报（Scribe → Human）**：Scribe 在 `story.md` 的重要推导 / 必要约束中显式列出 **候选 Story 列表 + 候选匹配依据 + 推荐处置方向**（new / supersede / duplicate / related）；**不下结论**，不下"已自动拒绝"。

4. **Story 处置（Human）**：在现有 Go / Park / No-Go gate，Human 显式裁决候选（accept all / waive duplicates / declare superseded / unrelated）；Runtime 持久化 `story_disposition{actor, reason, target_story_id, candidate_story_id, evidence, revision, disposition}`。

5. **Spec 起草前与起草中查询（B 路径 — Sage 上下文）**：Sage 在 draft `spec.md` 之前与修订过程中，program boundary 提供 `requirement_registry.query({spec_namespace, scope, topic, stage}) -> candidates[]`；Sage 将 superseded / duplicate / conflict 候选作为 **review context** 写入 spec.md 的"重要约束 / 候选"区块，**只用于提示作者与 reviewer**，**Sage 不得基于候选单独退役、改写或标记既有 requirement**。

6. **Spec review 阶段再查询（B 路径 — Lex 独立再查询）**：在 Sage 提交修订后、Runtime 调度 Lex 之前，Runtime 重新调用 `requirement_registry.query` 并把同一批候选连同最新的 spec.md revision 一并交给 Lex；Lex **不引用 Sage 的候选上报作为依据**，独立对当前 spec revision 做格式/语义 review 并把 superseded / duplicate / conflict 候选纳入 **independent review 输出**，每条候选给出 Lex 自己的判定依据（spec_namespace / revision digest / evidence）。

7. **Requirement 候选处置（Human）**：在现有 requirements approval（M-LOCK-1）与后续 M-LOCK gate，Human 显式裁决（accept / supersede prior / declare duplicate / declare conflict / waive gate block）；Runtime 持久化 `requirement_disposition{actor, reason, target_requirement_id, candidate_requirement_id, spec_namespace, evidence, revision, disposition}`。

8. **Registry 原子写入（Runtime）**：当 M-LOCK-1 完成、Spec 进入 locked 状态后，Runtime 原子或带可审计 evidence 把以下 identity / relationship 写入 Requirement Registry：`spec_id`, `spec_revision_digest`, `requirement_id`, `requirement_kind (FR|NFR)`, `requirement_revision_digest`, `spec_namespace`, `valid_status`, 以及该 Spec 相对 `registry_baseline` 的 admission record digest。Story Registry 接受 M-STORY 交接产出的 `story_id` + `story_revision_digest` + `bound_spec_id` + `release_id` + 对应 admission record digest。写入身份与 cutover identity 一并持久化，使历史/审计能反向确认哪些条目属于新 admission 合同下的产物。

9. **跨 Spec 引用语法**：在 Spec / Acceptance / Architecture / Interfaces 任意位置引用 **本 Spec 之外**的已批准 requirement 时：
   - 当局部 ID 在当前 Spec 上下文中无歧义：保持 `FR-NNNN` 短形式。
   - 当存在跨 Spec 歧义（同名 ID 出现在多个已批准 Spec）：使用 `FR-NNNN@<spec-id>` 限定形式，例如 `FR-2200@v0.14-002`。
   - Lex 的格式验收须解析两种语法，且不强制限定语法（不允许把短形式自动升级为限定形式）。

10. **阻断与豁免**：当候选未被 Human 处置时，Runtime 必须阻断对应 gate 推进；当 Human 提交豁免时，豁免遵守 v0.14.1 在 M-SPEC 阶段 **co-designed 的 Story/Requirement disposition 与 waiver 记录合同**（actor / reason / scope / target evidence digest / revision / recheck condition），该合同**复用** v0.14-001 / v0.14-002 / v0.14-003 已经确立的 bounded waiver 记录原则与不可豁免不变量集合（包含 v0.14-001 NFR-0100 原子性 / CAS、v0.14-003 FR-2700 waiver 字段、v0.14 BS-08 不可 waiver 不变量），但 **v0.14.1 之前并不存在完整覆盖 Story/Requirement disposition 域的 schema**——该 schema 是本 Story 在 M-SPEC 中与 disposition 路径共同设计后才生效的。豁免本身不可用于 requirements approval / M-LOCK / identity / CAS / artifact freshness / Agent 自批 / admission record digest / cutover identity。

11. **跨 Spec 引用解析**：Lex / Runtime 解析 `FR-NNNN@<spec-id>` 时只信任 Requirement Registry 中该 `(spec_id, requirement_id, requirement_kind)` 的最新 locked revision digest；解析失败 → 拒绝并报告 `REGISTRY_REQUIREMENT_NOT_FOUND`，不静默回退到短形式。

- **继续/返回**：
  - 处置后：当前 gate 继续；候选若被判定为 duplicate，源 / 目标 Spec 的关系被 Runtime 写入 Registry；后续起草可查询到。
  - 阻断：Human 在原 gate 上答复或新建 discussion；候选未处置前对应 task 不消耗 review 轮次。
  - 取消 / 退出：现有 Park / No-Go / return-upstream 路径不变；Registry 不会因 Park / No-Go 删除已批准 identity，仅记录 `parked_release` / `no_go_release` 关系。

### 2.2 行为种子

只提取 Sage 需要继续展开的用户结果和重要边界，不枚举普通微交互。

### BS-01 Registry 以 cutover identity 为分界初始化

- EARS: `WHEN v0.14.1 部署阶段 Runtime 创建 registry_baseline digest 与 registry activation record, THE 系统 SHALL 以该 baseline 为时间分界只索引 Runtime identity / creation / admission record 严格晚于该 baseline 且通过 v0.14.1 新 admission 合同的 Spec；历史 / cutover 当日草稿 / 未以新合同重新进入的资产 SHALL 保持可作为普通文件检索，但 SHALL NOT 被注册或回填`
- 来源: Human 决策 / 重要推导
- 说明: 把"部署时为空"具体化为可被引用的可审计 digest，避免 wall-clock-only 判定；与 v0.14-001 NFR-0100 / v0.14-003 FR-2700 的 CAS / record 类合同同构。

### BS-02 Scribe 上报疑似重复 Story 候选

- EARS: `WHEN Scribe 在 M-STORY 起草 story.md 之前, THE 系统 SHALL 只允许 Scribe 通过 program boundary 查询 Story Registry 并产出疑似重复候选清单，Scribe SHALL NOT 更新 Registry，SHALL NOT 自行裁定候选状态`
- 来源: Semantic 边界 / Human 决策
- 说明: Scribe 上报而非裁决；保持 Scribe 的语义调研者身份与 Runtime 的权威职责分离。

### BS-03 Sage / Lex 角色分离：Sage 上报为 context，Lex 独立再查询

- EARS: `WHEN Sage 在 M-SPEC 起草 spec.md 之前或修订过程中, THE 系统 SHALL 允许 Sage 通过 program boundary 查询 Requirement Registry 并把 superseded / duplicate / conflict 候选作为 review context 写入 spec.md；Sage SHALL NOT 仅基于候选对既有 requirement 做退役 / 改写 / 标记；WHEN Sage 提交修订后 Runtime 调度 Lex 之前, THE 系统 SHALL 重新执行 requirement_registry.query 并把当前 spec revision 与查询结果一并交给 Lex；Lex SHALL 独立判断并把候选纳入自己的 independent review 输出，且 SHALL NOT 引用 Sage 的候选上报作为判定依据；只有 Human 在 requirements approval / M-LOCK gate 显式处置后，对应 requirement 才被允许进入 locked 状态`
- 来源: Semantic 边界 / Human 决策 / 重要推导
- 说明: 防止"重复就静默退役"，同时保留 Lex 的独立 review 价值；候选的最终处置权属 Human，不在任何 Agent。

### BS-04 Registry 是索引而非内容事实源

- EARS: `THE 系统 SHALL 把 Story Registry 与 Requirement Registry 视为已批准 identity / relationship 的索引；任何 Registry 条目 SHALL NOT 取代 story.md / spec.md / acceptance.md 的 canonical 内容；Registry 查询失败 SHALL NOT 自动从 spec.md 文本回填`
- 来源: Semantic 边界 / 重要推导
- 说明: 避免形成"第二事实源"导致 split-brain；与 v0.14-002 FR-0700 Machine Contract Registry 同构的 program-owned registry 精神一致，但作用域不同（schema 治理 vs. Story/Requirement 索引）。

### BS-05 候选阻断 gate + v0.14.1 disposition 豁免路径

- EARS: `WHEN 候选未被 Human 显式处置, THE Runtime SHALL 阻断对应 gate 推进且 SHALL NOT 消耗 review 轮次；WHEN Human 提交豁免, THE 系统 SHALL 遵守 v0.14.1 在 M-SPEC 阶段 co-designed 的 Story/Requirement disposition 与 waiver 记录合同，并 SHALL NOT 把豁免用于 requirements approval / M-LOCK / identity / CAS / artifact freshness / Agent 自批 / admission record digest / cutover identity`
- 来源: Human 决策 / 重要推导
- 说明: "疑似必须阻断"是硬规则；豁免合同沿用 v0.14 bounded waiver 记录原则，但 Story/Requirement disposition 域本身的 schema 由本 Story 在 M-SPEC 中与 disposition 路径共同建立，**不是**对 v0.14 既有 waiver schema 的简单复用。

### BS-06 跨 Spec 限定引用的产品语义

- EARS: `THE 系统 SHALL 接受 FR-NNNN 短形式与 FR-NNNN@<spec-id> 限定形式；THE 系统 SHALL 只在引用方上下文存在跨 Spec 歧义时推荐使用限定形式；THE 系统 SHALL NOT 自动把短形式升级为限定形式`
- 来源: Human 决策 / 重要推导
- 说明: parser 不强制限定语法；产品语义优先，避免给作者增加不必要的格式负担。

### BS-07 限定引用解析权威性

- EARS: `WHEN Lex 或 Runtime 解析 FR-NNNN@<spec-id>, THE 系统 SHALL 只信任 Requirement Registry 中该 (spec_id, requirement_id, requirement_kind) 的最新 locked revision digest；解析失败 SHALL 返回 REGISTRY_REQUIREMENT_NOT_FOUND 并拒绝，不静默回退`
- 来源: 重要推导
- 说明: 限定语法必须真正消除歧义；解析失败必须显式，避免"看上去对，事实错"。

### BS-08 Registry 原子写入与可审计 evidence

- EARS: `WHEN Spec 进入 M-LOCK-1 或 Story 阶段交接产出最终 digest, THE Runtime SHALL 原子或带可审计 evidence 写入对应 Registry 条目；写入 SHALL 绑定 spec_revision_digest / story_revision_digest / actor / 时间戳 / 关系类型 / admission record digest / cutover identity；写入 SHALL NOT 修改既有条目的 locked revision`
- 来源: 重要推导 / v0.14-001 NFR-0100 / v0.14-003 FR-2700
- 说明: 复用 v0.14 原子性与 CAS 合同；registry 写入是只追加 + 关系更新，不修改历史 digest；cutover identity 写入使历史/审计能反向确认哪些条目属于新 admission 合同下的产物。

### BS-09 Wiki 显式排除

- EARS: `THE 系统 SHALL NOT 把 Story Registry 或 Requirement Registry 暴露给 Wiki 生成 / 消费路径；THE 系统 SHALL NOT 在 v0.14.1 引入 Wiki 生成或消费接口`
- 来源: Human 决策 / Out-of-Scope
- 说明: 把 Wiki 与 Registry 解耦，避免 Wiki 引入第二个事实源问题。

## 3. 范围、约束与例外

### 3.1 必须保持的产品约束

- **Registry 不取代内容**：Story/Spec/Acceptance 的 canonical 文件仍是内容事实源；Registry 只索引 identity 与已批准 relationship。
- **以 cutover identity 为分界**：Registry 仅覆盖 Runtime identity / creation / admission record 严格晚于 `registry_baseline` digest 且通过 v0.14.1 新 admission 合同的 Spec 与其 FR/NFR；历史 Story / Spec 保持可作为普通文件检索。即便 cutover 之前的资产被外部修改或被引用，只要它们没有以新 admission 合同重新创建为 post-cutover Spec，就不被 Registry 索引。
- **不与 v0.14 Runtime 冲突**：Registry 的写入与查询必须遵守 v0.14-001 NFR-0100（原子性 / CAS / 竞争请求）；豁免遵守 v0.14.1 在 M-SPEC 阶段 co-designed 的 Story/Requirement disposition 合同，复用 v0.14 bounded waiver 记录原则；不可 waive 检查清单遵循 v0.14-003 FR-2700 + v0.14 BS-08（requirements approval / M-LOCK / identity / secret / CAS / artifact freshness / Agent 自批 / release identity mismatch 永不可 waive），并在本 Story 上额外扩展为 admission record digest / cutover identity 也永不可 waive。
- **Runtime 是唯一权威写入方**：Scribe / Sage / Lex / Prism 通过 program boundary 查询 Registry；不得直接写入；写入仅由 Runtime 在 gate 通过后触发。
- **跨 Spec 引用解析权威**：限定形式解析只信任 Registry；解析失败必须显式拒绝，不静默回退。
- **Role boundary**：Sage 的 Registry 查询只产出 review context；Lex 在 review 阶段独立再查询并独立判定；Human 在 requirements approval / M-LOCK gate 显式处置。任何 Agent 不得基于 Registry 候选单独退役、改写或标记既有 requirement。

### 3.2 非常规要求

- **"阻断候选 + Human 豁免"模型（v0.14.1 首次引入于 Story/Requirement 级别）**：v0.14-003 FR-2700 既有 waiver 体系原本用于 program 检查的失败（security findings / trace freshness / release identity 等），其 bounded waiver 字段（actor / reason / scope / candidate / expiry / policy digest）作为本 Story 记录原则的复用基线。但 v0.14.1 之前**并不存在覆盖 Story/Requirement disposition 域的 schema**——本 Story 在 M-SPEC 阶段与 disposition 路径共同设计该域的 disposition 字段集、不可豁免清单扩展项与 audit 记录形态，并保留 v0.14 BS-08 既有不可 waive 不变量。这是有意偏离"自动去重"的常规做法，以保留 Human 在产品边界上的最终决定权。
- **Registry 作为"只追加索引"**：与 Machine Contract Registry（v0.14-002 FR-0700）同构的程序拥有、版本化、可发现的设计原则，但本 Registry 不存在 `active / candidate / retired` 三态；它只有一个 `locked` 状态，且条目只追加。这是与 `louke.machine-contract-registry` 的明确差异点，避免误用。
- **Lex 独立再查询而不是引用 Sage 输出**：与 Sage/Lex 在 v0.14-001 既定的"独立评审"合同一致，但本 Story 把"独立"具体化为对 Registry 的独立再查询与对候选的独立判定，避免形成"Sage → Lex 单向引用"造成的审查依赖链。

### 3.3 Out-of-Scope

- **Wiki 生成 / 消费**：v0.14.1 不把 Registry 接入 Wiki，不引入 Wiki 数据源或生成器；v0.11-001 已实现的 Wiki 由后续 Story 单独设计其与 Registry 的耦合。

## 4. 重要推导与证据

### D-01 Registry 应为 program-owned 的 Runtime-native 索引

- **结论**：Story Registry 与 Requirement Registry 由 Runtime 拥有，作为 program 边界；Scribe / Sage / Lex 只读，Human 通过 gate 裁决。
- **依据**：v0.14-002 FR-0700 Machine Contract Registry（程序拥有、版本化、可发现、不接受 instance 内嵌替代 schema）；v0.14-001 NFR-0100 原子性与 CAS 合同；v0.14-001 BS-02 权威 program result 边界。
- **影响**：决定 Registry 接口位置与可发现性，决定不暴露给用户作为独立 UI 入口；写入由 Runtime 在 gate 通过后原子触发。

### D-02 Scribe / Sage / Lex / Human 的职责切分与候选处置权属

- **结论**：Scribe 在 Story 起草阶段只产生"疑似重复候选清单 + 推荐处置方向"；Sage 在 Spec 起草与修订过程中只产生"疑似 supersede / duplicate / conflict 候选（作为 review context）"；Sage 不得基于候选单独退役 / 改写 / 标记既有 requirement。Lex 在 review 阶段独立重新查询 Registry 并独立判定，不引用 Sage 的候选上报作为依据。Scribe / Sage / Lex 不得更新 Registry。Human 在 requirements approval / M-LOCK gate 显式处置后才放行锁定。
- **依据**：Human 明确 Semantic 边界；v0.14-001 三层职责划分（program 自动化 / semantic 理解 / Human 决策）；v0.14-001 既定 Sage/Lex 独立评审合同。
- **影响**：决定 agent prompt 约束；决定 disposition 必须由 Human 显式做出；决定 Lex 在 review 阶段需要独立 query 而不是缓存 Sage 的查询结果。

### D-03 跨 Spec 限定语法只在歧义场景使用

- **结论**：`FR-NNNN@<spec-id>` 是 *opt-in* 限定语法；Lex 不强制、不自动升级；解析时只信任 Registry。
- **依据**：Human 明确 "Qualified reference should use Spec namespace only when cross-Spec ambiguity exists"；"local references remain short"；v0.13.1-002 已用 `AC-FR1510-01@v0.13.1` 验证过版本限定语法可读性。
- **影响**：决定 Lex parser 的语法设计（宽松）；决定 Sage 提示在何种条件下提示使用限定形式；不要在 Story 中锁定具体正则。

### D-04 v0.14.1 disposition / waiver 合同是本 Story 在 M-SPEC 共同设计的新合同，复用 v0.14 bounded waiver 记录原则

- **结论**：Story/Requirement disposition 域的 schema 与不可豁免清单扩展项由本 Story 在 M-SPEC 阶段与 disposition 路径共同设计。它**复用** v0.14-003 FR-2700 的 bounded waiver 记录原则（actor / reason / scope / target / candidate / expiry / policy digest 等字段形态）以及 v0.14 BS-08 不可 waive 不变量集合；并在本 Story 上额外扩展 admission record digest / cutover identity 永不可 waive。**v0.14.1 之前并不存在覆盖 Story/Requirement disposition 域的完整 schema**——Sage 在 M-SPEC 起草该合同时应把它作为新合同而非对既有 waiver schema 的简单复用，避免再次出现 v0.14 security workflow 的 program 检查 vs. semantic 候选之间的 schema 边界错位。
- **依据**：v0.14-003 FR-2700 Retry、Waiver 与取消（bounded waiver 字段、不可 waive 清单）；v0.14-001 BS-08 / flow.md L268 不可 waive 不变量；Human 明确"豁免不引入新机制"但同时明确 disposition 域的记录形态需在 v0.14.1 中定义。
- **影响**：避免发明第二套豁免机制造成规范扩散；要求 M-SPEC 中把 disposition 合同作为本 Story 的 co-design 产物；保留 v0.14 既有审查与审计能力。

### D-05 Registry 是只追加索引，不存在 active/candidate/retired 三态

- **结论**：Story Registry 与 Requirement Registry 的条目只有 `locked` 状态；supersede / duplicate / conflict 通过 *关系* 表达，不修改源条目。
- **依据**：本 Story 的范围（已批准 identity 的索引）；与 Machine Contract Registry（v0.14-002 FR-0700，schema 治理）区分；v0.14-001 NFR-0100 原子性。
- **影响**：决定 Sage 在 M-SPEC 中不应假定 Registry 条目存在 lifecycle 状态机；Registry 接口设计只暴露 identity + relationship + locked_digest。

### D-06 v0.14.0 发布独立推进

- **结论**：本 Story 不影响 v0.14-001 / v0.14-002 / v0.14-003 的当前发布工作；不修改既有 locked artifact；v0.14.0 release cutover 行为由既有 cutover-checklist 决定。
- **依据**：Human 明确 "Continue current v0.14.0 release work separately"；v0.14-001 m-lock-1-decision.json 已固化 24 个 requirements approval 与 locked contract digest。
- **影响**：本 Story 只在 v0.14.1 中实现，Registry 不回填 v0.14.0 的内容；既有助于测试隔离，也明确范围。

### D-07 Registry cutover identity 必须是 deterministic digest 而非 wall-clock

- **结论**：Registry 部署时为空由"Runtime 创建的 `registry_baseline` digest + registry activation record"作为可被审计、可被引用的 cutover identity 表达；只有 Runtime identity / creation / admission record 严格晚于该 baseline 且通过 v0.14.1 新 admission 合同的 Spec 才进入 Registry。wall-clock 比较不足以保证"何时起算"的可审计性，且无法在重启 / 时钟漂移 / 多次部署场景中保持稳定。
- **依据**：Human 决策"从部署起构建索引"；v0.14-001 NFR-0100 要求 record identity 可比、可审计；v0.14-001 m-lock-1-decision.json 用 digest 绑定 release identity 的既有先例。
- **影响**：决定 Registry 初始化必须把 cutover identity 持久化为 manifest / activation record 字段，并被所有后续条目引用；决定历史资产即便被外部修改也不被回填，除非以新 admission 合同重新创建为 post-cutover Spec。

- **既有能力**：

  - v0.14-001 BS-08 / FR-2700 不可 waive 不变量已实现；本 Story 在其上扩展 admission record digest / cutover identity 永不可 waive。
  - v0.13.1-002 的 `AC-FR1510-01@v0.13.1` 语法已验证限定引用可读性；本 Story 复用该语法形式（FR / NFR 同样适用）。
  - v0.14-001 m-lock-1-decision.json 已示范用 digest 绑定 release identity 的做法；本 Story 复用为 cutover identity 与 admission record digest 的表达方式。
  - 未发现已有 Story / Spec 覆盖 Story Registry 或 Requirement Registry。
- **冲突**：
  - 无与既有产品的实质冲突。本 Story 只在 v0.14.1 中新增身份与关系索引；不修改既有 Story / Spec 内容、不修改 Machine Contract Registry（v0.14-002 FR-0700）、不修改既有 bounded waiver 字段（v0.14-003 FR-2700）。
- **重要风险**：
  1. **重复 / 冲突阻断引入额外 review latency**：每次新 Spec 起草都增加一次 Registry 查询 + 候选处置；Human 必须显式回答。建议把候选清单与现有 Go / Park / No-Go / requirements approval / M-LOCK gate 并入同一交互面，避免新增独立 UI。
  2. **Sage → Lex 单向引用风险**：若 Lex 直接复用 Sage 的 Registry 查询结果而不独立再查询，会形成审查依赖链，与 v0.14-001 既定的 Sage/Lex 独立评审合同冲突。本 Story 要求 Runtime 在 Lex 调度前重新执行 query，并由 Lex 独立判定，避免依赖 Sage 上报。
  3. **限定语法过度规范化风险**：若 Lex 强制所有跨 Spec 引用使用 `@<spec-id>`，作者负担与歧义解决收益不匹配；本 Story 明确 Lex 不强制、不自动升级。
  4. **Registry 事实源分裂风险**：若未来 Wiki 或其他消费者把 Registry 当内容源使用，会与 story.md / spec.md 出现分歧；本 Story 显式排除 Wiki 与 Registry 的耦合，但需在 cutover 后保持该约束。
  5. **disposition 合同错位为 waiver schema 复用风险**：若 M-SPEC 把 Story/Requirement disposition 域直接套用 v0.14-003 FR-2700 既有 program 检查 waiver 字段而不重新审视 semantic 候选与 program 检查之间的 schema 边界，会引入新的不一致（既有 v0.14 security workflow 的 waiver 字段针对 finding / candidate binding，本 Story 的 disposition 字段针对 Story / Requirement identity / relationship）。M-SPEC 应把 disposition 合同作为本 Story 的 co-design 产物，并保留 v0.14 BS-08 不可 waive 不变量集合。
  6. **cutover identity 引入额外入口风险**：若 cutover identity 仅以 wall-clock 而非 digest 表达，重启 / 时钟漂移 / 多次部署会导致"何时起算"不可审计，且无法在 admission record digest 中稳定引用。本 Story 要求 Runtime 创建 `registry_baseline` digest + activation record，把"部署时为空"具体化为可被引用的可审计记录。
  7. **v0.14.0 不受影响的风险**：若 Runtime 实现错误地把 v0.14.0 内容回填到 Registry，会违反 Human 决策 1；本 Story 要求 Runtime 在初始化时只承认 Runtime identity / creation / admission record 严格晚于 `registry_baseline` digest 且通过新 admission 合同的 Spec，并通过 admission record digest 在每条条目上显式区分。
- **分流建议**：**Go（Agent 建议）**。依据：所有产品决定已由 Human 在原始输入中固化；既有 v0.14 能力（v0.14-002 FR-0700 program-owned / 版本化 / 可发现、v0.14-003 FR-2700 bounded waiver 字段形态与不可 waive 不变量集合、v0.14-001 NFR-0100 原子性与 CAS、v0.13.1-002 限定语法可读性、v0.14-001 m-lock-1-decision.json digest 绑定 release identity 的先例）足以承载本 Story 的设计基线；本 Story 在 v0.14.1 中以 cutover identity 为分界新增身份与关系索引，Out-of-Scope 明确，Wiki 显式排除；本 Story 不修改 v0.14.0 locked artifact，符合 Human 决策 4。
