## B. P0 — 阻塞发布

### P0-1: 定位算法 — anchor-hash 未定义

**问题**：`lk discuss start` 依赖 `--anchor-hash <hash>` 来定位讨论插入点，但 spec 没有定义：

1. **什么是"被评论内容"**：是被讨论段落的前一行？整个段落？还是 `### FR-XXXX` 标题下的全部内容？
2. **归一化规则**：hash 之前做什么处理？strip 首尾空格？去 markdown 格式（`**`、`` ` ``）？转 lowercase？
3. **hash 算法**：SHA256 的输入是单行文本还是多行拼接？拼接方式？
4. **`--anchor` 生成命令缺失**：SKILL.md L135 说"用 `--anchor` 标志在文章中重新生成"，但 `lk discuss` 三个子命令里都没有 `--anchor` 子命令或标志。agent 怎么获取 anchor-hash？手动调 sha256sum？
5. **碰撞处理**：如果文件中两段内容归一化后相同（例如同一个短句子 "需要确认" 出现在两个 FR 下面），anchor-hash 碰撞怎么办？
6. **插入位置模糊**：当 anchor 跨多行（一个段落 5 行），"插入到定位行的下一行"是哪个下一行？段落后？还是第一行后？

**影响**：没有定位算法，`lk discuss start` 无法实现。这是整个 write path 的基础。

**建议**：新增 FR-0025 "定位算法规格"，定义：
- anchor 范围 = 根评论 `>` 上方最近的非空、非 `>` 行（单行锚定）
- 归一化 = strip 首尾空白 + Unicode NFC 标准化（不改大小写、不去 markdown 格式）
- hash = SHA256(normalized_text) 的 hex digest 前 16 字符（够用且人类可读）
- 碰撞处理 = 在 anchor-hash 后追加 `:N` 序号（第 N 个同 hash 段落）
- 新增 `lk discuss anchor --file <path>` 子命令，输出文件中所有可锚定段落及其 hash

### P0-2: thread_id 与定位算法 — 完全设计

**问题**：spec FR-0020 AC-5 将 4 元组做 SHA256 生成 thread_id，这是一个根本性错误。SHA256 把结构化定位信息压成不可逆的不透明串，行号漂移一个字符就导致整体失效。正确做法是：thread_id 作为简单标识，定位字段独立传递给 parser，parser 用**行号 + Levenshtein 双重比对**定位 thread。

#### 设计原则

1. **thread_id 是句柄，不是定位器**：thread_id 只需要在文件内唯一，不需要编码位置信息
2. **定位字段是 query 输出的一部分**：agent 从 query 结果中拿到定位字段，传给 reply/modify 命令
3. **行号是 hint，Levenshtein 是兜底**：行号快速跳转，编辑距离容忍小幅变更
4. **不依赖 SHA256 做匹配**：SHA256 精确匹配太脆弱，编辑距离才是正确的相似度度量

#### 5 元组定位字段

每个 thread 在 query 输出中包含以下定位字段：

| 字段          | 类型   | 含义                                       | 用途                                                 |
| ------------- | ------ | ------------------------------------------ | ---------------------------------------------------- |
| `total_lines` | int    | 创建时文件总行数                           | 估算行号偏移量 `delta = current_total - total_lines` |
| `anchor_line` | int    | 被评论内容的行号                           | 快速跳转的起始 hint                                  |
| `anchor_text` | string | 被评论内容的原文（归一化后）               | Levenshtein 匹配的基准文本                           |
| `root_line`   | int    | 根评论 `>` 行的行号                        | 快速跳转的目标 hint                                  |
| `root_text`   | string | 根评论内容原文（speaker + body，归一化后） | Levenshtein 匹配的基准文本                           |

**归一化规则**：strip 首尾空白 + 合并连续空白为单空格 + Unicode NFC。不改大小写，不去 markdown 格式。

#### thread_id 格式

thread_id 使用简单的自增序号（per file），格式 `T-{NNN}`，如 `T-001`、`T-002`。

parser 维护一个文件内的 thread 计数器。新 thread 的 id = 当前最大序号 + 1。thread_id 仅用于 query/reply/modify 命令中引用，不承载定位功能。

#### query 输出格式

```json
[
  {
    "thread_id": "T-001",
    "total_lines": 230,
    "anchor_line": 45,
    "anchor_text": "用户用邮箱 + 密码登录。密码至少 8 位。",
    "root_line": 47,
    "root_text": "Sage: What should happen when depth is zero?",
    "initiator": "Sage",
    "status": "open",
    "last_speaker": "Aaron",
    "reply_count": 2,
    "snippet": "What should happen when depth is zero?"
  }
]
```

Agent 在第一轮拿到这些字段，第二轮阅读并决策，第三轮把定位字段传回 parser。

#### 查找算法（4 级降级）

当 agent 调用 `lk discuss reply --thread-id T-001 --anchor-line 45 --anchor-text "..." --root-line 47 --root-text "..."` 时：

```
Step 0: 计算行号偏移
  current_total = len(file.readlines())
  delta = current_total - stored_total_lines
  adjusted_anchor = anchor_line + delta
  adjusted_root = root_line + delta

Level 0 — 精确命中（最快路径）:
  检查 adjusted_anchor 行: normalize(text) == anchor_text ?
  检查 adjusted_root 行:   normalize(text) 包含 speaker-tag 且 root_text 匹配 ?
  → 两者都匹配 → 命中 ✓
  → 适用场景: 文件未变，或仅在该 thread 之后有增删

Level 1 — Levenshtein 窗口搜索（行号漂移 + 内容微改）:
  搜索范围 = adjusted_anchor ± max(|delta| + 5, 10) 行
  对范围内每行计算: edit_distance(normalize(line), anchor_text)
  找到距离最小的候选行 → 在其下方扫描 blockquote 块
  对块内根评论计算: edit_distance(normalize(root), root_text)
  → anchor 距离 ≤ 阈值 AND root 距离 ≤ 阈值 → 命中 ✓
  → 阈值 = max(5, len(text) * 0.2)  # 容忍 20% 的编辑

Level 2 — 仅根评论定位（锚点被大幅修改或删除）:
  全文扫描所有根评论（depth=1 的 `>` 行）
  对每个根评论计算: edit_distance(normalize(root), root_text)
  → 距离最小的候选，且 speaker 匹配 → 命中 ✓
  → 此时 anchor 信息已不可靠，更新 thread 的 anchor 字段

Level 3 — 未找到:
  返回 "thread not found"
  输出包含: thread_id, 最后已知位置, 建议操作（重新 query 获取新定位字段）
```

在工具找不到匹配点时，Agent应该依照 disucssion 语法，自己决定如何修改文件。

#### 为什么不需要担心 hash 碰撞

此方案**不使用 hash 做匹配**，改用 Levenshtein 距离。碰撞问题转化为：两个不同文本的编辑距离 ≤ 阈值。这在以下条件下极不可能：

- **两重定位**：anchor_text 和 root_text 必须同时模糊匹配到同一个 thread 的对应位置
- **空间约束**：Level 1 只在窗口内搜索，不是全文盲扫
- **内容长度**：anchor_text 和 root_text 通常 ≥ 10 个字符，编辑距离阈值内的随机碰撞概率极低

即使极端情况下 Level 1 误匹配，Level 2 的全文扫描 + speaker 验证也能兜底。

#### CLI 接口（5 个子命令）

```bash
# 1. query — 输出包含定位字段的 thread 列表
lk discuss query --file <path> [--initiator <agent>] [--blocker <agent>] [--status <s>]

# 2. start — 创建新 thread，输出 thread_id 和定位字段
lk discuss start --file <path> --anchor-line <N> --speaker <agent> "<message>"
# parser 自动计算 anchor_text（取 anchor-line 的归一化内容）

# 3. reply — 追加回复，agent 传回定位字段
lk discuss reply --file <path> --thread-id <id> \
  --anchor-line <N> --anchor-text "<text>" \
  --root-line <N> --root-text "<text>" \
  --speaker <agent> "<message>"

# 4. edit — 修改自己某条评论的内容（仅限原作者）
lk discuss edit --file <path> --thread-id <id> \
  --anchor-line <N> --anchor-text "<text>" \
  --root-line <N> --root-text "<text>" \
  --depth <N> --speaker <agent> "<new content>"
# parser 验证 speaker == 该评论原作者，否则拒绝

# 5. set-status — 修改 thread 状态（仅发起人可 RESOLVED，任意人可 REOPEN）
lk discuss set-status --file <path> --thread-id <id> \
  --anchor-line <N> --anchor-text "<text>" \
  --root-line <N> --root-text "<text>" \
  --status <resolved|reopen>
```

`--anchor-line`/`--anchor-text`/`--root-line`/`--root-text` 是**冗余定位参数**。`--thread-id` 是主键，定位字段辅助 parser 在文件变更后仍能定位。即使 agent 传了旧的定位字段，parser 的降级策略也能处理。

#### 边界情况

| 情况                            | 降级路径                        | 结果             |
| ------------------------------- | ------------------------------- | ---------------- |
| 文件无变化                      | Level 0 直接命中                | ✓                |
| 文件头部增删行（行号漂移）      | delta 修正 → Level 0 或 Level 1 | ✓                |
| 锚点内容小幅修改（改措辞/typo） | Level 1 Levenshtein 匹配        | ✓                |
| 锚点内容被大幅重写              | Level 2 靠 root_text 定位       | ✓（更新 anchor） |
| 锚点被删除但讨论还在            | Level 2 命中，标记 orphan       | ⚠️ orphan         |
| 整个讨论被删除                  | Level 3 not found               | ✗                |
| 文件重命名/移动                 | 不影响（基于文件内容）          | ✓                |

**建议**：
- spec FR-0020 替换 AC-5，写入上述完整定位算法
- 删除 "SHA256 4 元组" 描述，改为 "5 元组定位字段 + Levenshtein 降级查找"
- FR-0030 按三层架构重写，5 个子命令（query/start/reply/edit/set-status），签名含定位参数（见 P0-4）
- FR-0010 AC-1 中提到的 SKILL.md "7 处修正" 需包含此算法修正

### P0-3: is_ready / check-ready 门禁逻辑未迁移

2. Quote 状态：该 FR/NFR 下没有 open quote

新 spec（FR-0020）只提到 thread 状态（open/resolved/reopen），**完全没有 YAML 元数据维度**。同时 FR-0020 AC-7 说"现有 unit 切分 + YAML 解析保持兼容"，但没有描述新协议下 `is_ready` 的具体判定规则。

**影响**：

**建议**：FR-0020 新增一节 "is_ready 迁移"，明确：
- `lk discuss query --file <path> --check-ready` 或等效命令

### P0-4: `lk discuss` 架构与写操作语义不足

**问题 1：架构层次未定义**

FR-0030 说"新建 `louke/discuss.py`"，但没有定义模块层次。正确架构为三层：

```
lk sage quote-check / lk sage record-lock    ← Layer 1: Agent 专属命令（带特定流程语义）
       ↓ 内部调用
lk discuss query/start/reply/edit/set-status  ← Layer 2: 共享 CLI（所有 agent 和用户可用）
       ↓ 内部调用
louke/_tools/discuss.py                        ← Layer 3: 底层工具包（parser + 定位算法 + 写操作）
```

- Layer 3（`_tools/discuss.py`）：底层工具，包含 parser、定位算法、文件写操作。可被 Layer 1 和 Layer 2 调用。原 `quote_parser.py` 的解析逻辑合并至此（或 discuss.py 调用重写后的 quote_parser）。
- Layer 2（`lk discuss`）：user-facing 顶级命令，5 个子命令。CLI 入口在 `louke/__main__.py` 注册。
- Layer 1（`lk <agent> <command>`）：agent 专属命令，如 `lk sage quote-check` 做门禁、`lk sage record-lock` 做锁定。内部调用 `_tools/discuss.py` 的 API。

**问题 2：写操作语义不足**

FR-0030 只定义了 `start` 和 `reply` 两个写操作。完整写操作应为 3 个（`start`/`reply`/`edit`）+ 1 个状态操作（`set-status`），且以下语义未定义：

1. **`start` 的插入逻辑**："在内容后下一行插入"——如果该位置已有其他讨论呢？新讨论插在已有讨论的前面还是后面？
2. **`reply` 的追加逻辑**："在 thread 末尾追加"——末尾是指最后一行 `>>>` 的下一行？如果 thread 后面紧跟另一个 thread 的 `>` 呢？（需要在两个 `>` block 之间插入空行）
3. **`edit` 的内容替换**：替换指定 depth + speaker 的评论内容。如果新内容跨多行，需要保持 `>` 前缀和缩进层级一致。
4. **`set-status` 的权限**：仅发起人可设 RESOLVED，任意人可设 REOPEN。parser 需验证 `root speaker == 操作者` 才允许 RESOLVED。
5. **并发安全**：两个 agent 同时 `reply` 到同一个 thread，或同时 `start` 在同一 anchor 下，怎么处理？AC-5 说"重新 parse 验证状态机一致"，但没有说 parse 失败怎么办——回滚？重试？报错？
6. **空行分隔**：blockquote 之间需要空行才能被 CommonMark 解析为独立块。所有写操作是否自动插入空行？

**建议**：FR-0030 重写，定义：
- 三层架构（如上）
- 5 个子命令的完整签名（见 P0-2 CLI 接口）
- `start` 总是在 anchor 段落后的第一个空行之后插入；如果该位置已有讨论，插在已有讨论之后（同 anchor 下多个 thread 按时间顺序排列）
- `reply` 在 thread 最后一行之后插入，确保与下一个 `>` block 之间有空行
- `edit` 定位到 thread 内指定 depth+speaker 的评论，替换内容，parser 重新验证
- `set-status` 编辑根评论行的状态标记，验证发起人权限
- 所有写操作采用 file lock（`flock` 或 `.lock` 文件），parse 失败则回滚（删 tmp 文件，报错退出）

### P1-5: 旧状态标记（blocked-by-N / wontfix / superseded）静默删除

当前 `quote_parser.py` 支持 5 种状态：`[open]`、`✓ resolved`、`[blocked-by-N]`、`[wontfix]`、`[superseded]`。

SKILL.md 只定义了 3 种：`[open]`（隐式）、`[RESOLVED]`、`[REOPEN]`。

`[blocked-by-N]`（FR 间依赖）、`[wontfix]`（不修复）、`[superseded]`（被替代）全部丢失。spec 没有说明原因，也没有迁移方案。

**影响**：
- 现有 spec.md 中如果有 `[blocked-by-N]` 或 `[wontfix]` 的 quote，重写后的 parser 无法处理
- "blocked-by" 语义在多 FR spec 中有实际用途（FR-A 依赖 FR-B 的决定）

**建议**：明确声明废弃并在 FR-0020 加一行说明。如果确实不需要，在 "0.2 不收纳" 中列出 "旧状态标记 blocked-by-N / wontfix / superseded（简化为 open/resolved/reopen 三态）"。
