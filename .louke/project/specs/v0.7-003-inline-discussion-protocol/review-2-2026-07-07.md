## 新问题

### P0-NEW: FR-0060 `reopen` 阻塞逻辑有误

FR-0060 写道：

> 单元下**所有 thread 状态 ≠ open**（即全部 resolved 或 reopen 不算阻塞）

这意味着 `status == "reopen"` 的 thread **不阻止** FR ready。逻辑上是错的——thread 被 reopen 表示有人对结论不满意、讨论重新打开，FR 不应该视为 ready。

**修正**：ready 判定应为：

```
ready = yaml_resolved == "✅" AND 单元下所有 thread.status == "resolved"
```

即 reopen **算**阻塞。等价表述：`不存在 status == "open" 或 status == "reopen" 的 thread`。

**影响范围**：FR-0020 AC-4（is_ready 接口）、FR-0060 AC-1/AC-2、FR-0080 测试用例。

### P1-NEW-2: `lk discuss anchor` 子命令缺失

FR-0010 S7 提到 `--anchor` 标志未定义，暗示需要新增 `lk discuss anchor` 命令。但 FR-0030 的 5 子命令列表中没有 `anchor`。

**现状**：`lk discuss start --anchor-line <N>` 直接接受行号，不需要预计算 hash，所以 `anchor` 子命令可能不再必要。

**建议**：要么从 S7 中删除对 `anchor` 子命令的引用，要么在 FR-0030 中加上它（用途：列出文件中所有可锚定段落及其行号，方便 agent 选择 anchor_line）。

### P1-NEW-3: @mention 语法去留未声明

首次评审 P1-2 要求明确 @mention 是保留还是废弃。新版 spec 完全未提及。

**影响**：`_protocols/quote-dialogue.md` 将被删除（FR-0070），该文件是 @mention 语法的唯一定义。删除后 @mention 将没有规范来源。但 SKILL.md §5 示例中 Maestro 仍使用 `@Aaron` mention，§2.2 表格"评论内容"中也写了"支持 `@提及`"。

**建议**：在 FR-0010 或 FR-0020 中加一句声明：@mention 语法保留（parser 识别 `**@Speaker:**` 前缀）或废弃（示例中的 `@Aaron` 改为普通文本）。

## 细节问题（P2）

### P2-1: FR-0030 `louke/discuss.py` CLI 入口文件

FR-0030 说"louke/discuss.py 作 CLI 入口（不含实现，调 `_tools/discuss.py`）"。这增加了一个额外的薄文件。如果 CLI 注册在 `__main__.py`，这个文件可以省略，直接在 `__main__.py` 中 import `_tools/discuss.py`。不影响功能，但增加一层间接性。

### P2-2: 用户故事 §1 第 5 条

> agent 调 `lk discuss query --blocker <我的名字>` 看 3 类别（`unanswered` / `unresolved` / `awaiting_my_reply`）

这 3 个类别名在 spec 其他地方没有定义。query 命令的 `--blocker` flag 行为是"thread 最后一层不是该 agent 写的，且未 RESOLVED"，但 `unanswered` / `unresolved` / `awaiting_my_reply` 的区分逻辑未说明。建议在 FR-0020 或 FR-0030 中定义这 3 个类别。

### P2-3: FR-0080 测试数量
