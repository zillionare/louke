### 1. scenario-0200 日期 / 星期 + "昨天" 语义双重错误

**位置**：spec.md §2 scenario-0200（行 128-141）

scenario 声称：

```
# 周五 cron 没跑 (机器关机), 周六开机
[librarian.compact] 蒸馏窗口: [2026-07-02, 2026-07-04]  (catch-up 跨 3 天)
```

**错误 A — 星期不匹配日期**：2026-07-02 = Thursday、2026-07-03 = Friday、2026-07-04 = Saturday。scenario 标注"周五没跑"但实际周五是 07-03，不是 07-02。

**错误 B — 窗口上限违反 FR-0080 定义**：FR-0080 定义窗口为 `[last_distill, 昨天]`。当 cron 在周六（07-04）04:00 执行时，`date.today() = 2026-07-04`，`yesterday = 2026-07-03`。因此正确窗口应是 `[2026-07-02, 2026-07-03]`，不是 `[2026-07-02, 2026-07-04]`。scenario 把"昨天"写成了"今天"。

现有 `cmd_daily`（`librarian.py:293`）实现正确：`yesterday = (today - timedelta(days=1)).isoformat()`。spec scenario 与自己的代码实现不一致。

**连锁影响**：scenario-0200 的周日段也有问题。spec 写 `[2026-07-04, 2026-07-05]`，但正确应为 `[2026-07-03, 2026-07-04]`（如果上次 last_distill 更新为 07-03，周日 today = 07-05，yesterday = 07-04）。

**建议**：重写 scenario-0200，用正确日期或改用相对日期（"Day N"），并确认"昨天"语义与代码一致。

### 3. M2 模式下旧 `.compact-bundle.md` 被当作 map 输入（陈旧数据喂入 LLM）

**位置**：spec.md FR-0140.3 `cmd_rewrite` 实现（行 425-444）

`cmd_compact` 在 M2 模式下生成 `.compact-bundle-{YYYY-MM}.md` + `.compact-bundle-merged.md`，但 spec 没有说会删除之前 M0 遗留的 `.compact-bundle.md`。

`cmd_rewrite` 的 M2 检测逻辑：

```python
bundles = sorted(glob('.compact-bundle*.md'))
# bundles 包含: ['.compact-bundle.md', '.compact-bundle-2026-01.md', ..., '.compact-bundle-merged.md']
# 第一个 if 跳过 (len > 1)
# 第二个 if: merged 存在 → 进入 M2
for b in bundles[:-1]:  # map phase — 包含旧的 .compact-bundle.md!
    prompt = build_map_prompt(b)
    opencode_run(...)
```

旧的 `.compact-bundle.md` 被当作第 13 个 map 输入处理，产生一个多余的 mini-distillation，且内容与新 bundle 重叠 → reduce 阶段拿到重复信息。

**建议**：FR-0140.2 `cmd_compact` 进入 M2 时，应删除 `.compact-bundle.md`（如果存在）；或在 FR-0140.3 `cmd_rewrite` 的 map 循环中排除非 `-{YYYY-MM}` 格式的 bundle。

## P1 — 重要（设计缺口，实现前须澄清）

### 4. compact 不清理旧 bundle —— 磁盘累积 + 模式判定漂移

**位置**：FR-0140.2 `cmd_compact`（行 406-423）

`cmd_compact` 每次跑都写入新的 bundle 文件，但从不删除旧的。场景：

- Day 1: M0 → 写 `.compact-bundle.md`
- Day 2: 数据量增加但仍 M0 → 覆写 `.compact-bundle.md`（OK，同名覆写）
- Day 30: catch-up 跨 30 天 → M2 → 写 12 个月度 bundle + merged

但 Day 30 的 M2 不会清理 Day 29 的 `.compact-bundle.md`（如上 P0-3），且后续每个月的 M2 跑都不会清理上个月的月度 bundle。

**建议**：FR-0140.2 添加 bundle 清理逻辑 —— compact 开始时删除所有 `.compact-bundle*.md`（`.louke/wiki/` 下的临时文件，不影响 git tracked 内容）。这与 `compact` 的"准备 bundle"语义一致：bundle 是中间产物，不是持久数据。

### 5. wiki `decisions/`、`entries/`、`consolidated.md` 不在重写范围内

**位置**：FR-0080 `cmd_rewrite` + FR-0120 §2.3 permissions

现有 wiki 结构（实际 on disk）：

```
.louke/wiki/
├── pages/          (7 files — spec 聚焦)
├── decisions/      (11 ADR files — 008 编号重复!)
├── entries/        (7 legacy files — 迁移前产物)
├── consolidated.md (legacy — 最后整合 2026-05-23)
```

spec 的 rewrite 范围仅限 `pages/`。但：

- `decisions/` 有 11 个 ADR，其中 008 编号重复（`008-multi-ide-boards.md` + `008-bats-test-name-convention.md`）。LLM 整体重写时是否应整合？

- `consolidated.md` 是 legacy 整合文件。rewrite 后是否应删除/更新？

FR-0120 §2.3 的 edit 白名单也只列了 `pages/*.md` + `index.md` + `log.md` + `overview.md` + `.cache.toml`，没有 `decisions/`。

**建议**：在 §0 范围与边界中明确 decisions/ entries/ consolidated.md 的处理策略。至少应声明 `entries/` 为 deprecated（spec §0.3 不收纳），`decisions/` 是否纳入 rewrite 范围需要 Kilo 决定。

### 6. FR-0120 §2.3 edit 白名单与现有 Librarian.md 不一致

**位置**：FR-0120 §2.3（行 520-521）

spec 声称 edit 白名单为：`pages/*.md via rewrite + index.md + log.md + overview.md + .cache.toml`

现有 Librarian.md §2.3 白名单为：`pages/*.md`（via from-raw + distill）+ `index.md` + `log.md` + `overview.md` + `.cache`

差异：

| 项目 | 现有 Librarian.md | spec FR-0120 |
|---|---|---|
| pages/ 写入路径 | `from-raw` + `distill` | `rewrite`（整体重写） |
| `.cache` vs `.cache.toml` | `.cache`（泛指） | `.cache.toml`（精确） |
| `.compact-bundle*.md` | 不提及 | 不提及（但 compact 写 bundle 到 wiki/） |

Librarian 需要写 `.compact-bundle*.md` 到 `.louke/wiki/`，但 FR-0120 §2.3 的 edit 白名单不包含 bundle 文件。虽然 bundle 是 python 脚本（`cmd_compact`）写的，不是 LLM 的 `edit` 工具写的，但 spec 应明确这个区分。

**建议**：FR-0120 §2.3 补充说明 —— bundle 文件由 python 脚本写入（不受 `edit` 权限约束），LLM 的 `edit` 仅限 `pages/*.md` + `index.md` + `log.md` + `overview.md`。

### 7. `--model` vs `--model-from-config` 语义冲突

**位置**：FR-0140.4（行 448-463）

### 8. raw 条目缺 `date` 字段时的处理未定义

**位置**：FR-0080 `cmd_compact` 步骤 2（行 215）

compact 按 `date` 字段过滤 `[last_distill, 昨天]`，但 spec 没说不含 `date` 字段的 raw 条目怎么处理。现有 `cmd_from_raw`（`librarian.py:248-250`）的处理是：

```python
date_m = re.search(r'^date:\s*(\d{4}-\d{2}-\d{2})', content, re.MULTILINE)
file_date = date_m.group(1) if date_m else ''
if args.since and file_date and file_date < args.since:
    continue
```

即：无 date 的条目不会被过滤掉（`file_date = ''`，`'' and '' < since` = False），会被无条件包含。

**建议**：FR-0080 显式定义：无 `date` 字段的 raw 条目是 (a) 跳过 + warning，(b) 无条件包含，还是 (c) 报错。推荐 (a)。

## P2 — 建议（改进空间，不阻塞）

### 9. FR-0070 三原则第 3 条措辞模糊

**位置**：FR-0070（行 198）

> python 脚本只做机械工作，distillation 由 LLM（Librarian subagent）做：脚本不调 LLM API

"不调 LLM API"容易被理解为"不能发 HTTP 请求到 LLM endpoint"。实际 spec 意思是"不直接 import / 调用 OpenAI / Anthropic SDK"，而 shell-out 到 `opencode run` 是允许的。

**建议**：改为"脚本不直接调用 LLM SDK；LLM 推理通过 `opencode run --agent` CLI 入口完成"。

### 10. 缺少 quote-dialogue 语法交叉引用

**位置**：FR-0080 rewrite prompt（行 232、333）

spec 多处要求"每条 wiki 决策必须能从 raw 中找到依据（quote dialogue 语法）"，但没有引用定义此语法的 spec v0.4-004（quote-dialogue）。Librarian prompt 需要知道 quote dialogue 的具体格式。

**建议**：在 §6 关联文件或 FR-0080 中添加 v0.4-004 引用。

### 11. FR-0080 幂等性声明不够精确

**位置**：FR-0080（行 223）

> **幂等**：再跑一次 window 为空，零输出

严格来说：compact 跑完后 `last_distill = 昨天`，第二次跑的 window = `[昨天, 昨天]`（单日区间），不是"为空"。只有当该日没有新 resolved raw 时才是零输出。

**建议**：改为"幂等：再跑一次若无新 resolved raw，则 bundle 不变 + cache 不变"。

### 12. `cmd_lint` 现有实现只检查 `[[wikilink]]`，不检查 frontmatter

**位置**：FR-0110（行 280-288）

spec 说"保留现有检查 + 新增 frontmatter 必填字段检查"。但现有 `cmd_lint`（`librarian.py:140-190`）只检查 broken links 和 orphaned pages，没有 frontmatter 检查。Librarian.md §5.3 列了 4 个检查项（含 frontmatter + 重复页面），但代码没实现。

FR-0110 应明确这是**首次实现** frontmatter 检查，不是"保留"。当前措辞容易让实现者以为代码已有此功能。

### 13. `cmd_rebuild_index` 现有实现是扁平列表，不是"按类型 + 按日期"

**位置**：FR-0080 rewrite 步骤 5（行 242）

rewrite prompt 要求 LLM 调 `lk librarian rebuild-index` 重建 index.md。但现有 `cmd_rebuild_index`（`librarian.py:193-212`）只生成扁平列表（`- [[stem]] (path)`），不分类。Librarian.md §4 要求"按类型 + 按日期两个维度列出所有页面"。

这是已有 bug（代码与 agent prompt 不一致），v0.7-002 不修复但应知晓。

### 14. FR-0130 prompt 硬编码 bundle 路径为 `.compact-bundle.md`

**位置**：FR-0130 prompt（行 318）

```python
prompt = f'''
...
输入：
1. 读 {bundle}（含 raw 全文 + 现有 pages/ + 蒸馏指令）
```

`bundle` 变量是 `.compact-bundle.md` 的绝对路径。在 M2 模式下，`cmd_rewrite` 应传不同的 bundle 路径（月度 bundle 或 merged bundle）。FR-0140.3 的 `cmd_rewrite` 伪代码用 `build_map_prompt(b)` / `build_reduce_prompt(bundles[-1])`，暗示 prompt 是动态构建的，但与 FR-0130 的硬编码 prompt 矛盾。

**建议**：FR-0130 应声明 prompt 模板是基础版，FR-0140.3 的多模式 prompt 是扩展版。两者不应分别写死。
