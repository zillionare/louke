# v0.8-001 — Web Server 协作界面 — Test Plan

- **Spec ID**: v0.8-001-web-server
- **创建日期**: 2026-07-08
- **关联 acceptance**: `.louke/project/specs/v0.8-001-web-server/acceptance.md`
- **关联 interfaces**: `.louke/project/specs/v0.8-001-web-server/interfaces.md`

## 1. 立场与边界

### 1.1 黑盒声明

本测试计划只声明**从系统外部可观测**的测试方法，不依赖前端/后端内部实现细节。当前被测对象是一个面向团队内多人协作的 web server 及其浏览器界面，外部可观测出口限于：

- web server 启动入口、HTTP 端点、健康检查与退出码
- 浏览器中的页面结构与可交互 UI 行为
- 文档与 wiki 的持久化文件结果
- 模型绑定的可见状态与持久化结果
- 结构化日志、错误提示、冲突提示

### 1.2 不可观测对象

- 前端状态管理实现方式
- 编辑器组件内部数据结构
- 拖拽库、Markdown 解析库、协议解析器的内部实现
- 服务端内部缓存、队列、会话对象

> **可观测契约**：凡 acceptance 需要验证但当前不可直接观测的状态，必须在 `interfaces.md` 中由实现层提供外部可观测出口，而不是让测试窥探内部对象。

### 1.3 作伪模式

| #   | 作伪模式              | 典型表现                                            |
| --- | --------------------- | --------------------------------------------------- |
| 1   | 改断言迁就实现        | spec 要求拖拽绑定，测试改成“点击按钮也算通过”       |
| 2   | 只测静态渲染不测交互  | 页面看起来有卡片，但不验证切换、折叠、保存行为      |
| 3   | 只测 DOM 存在不测语义 | `data-testid` 存在就算通过，不验证正文/讨论主次对换 |
| 4   | mock 掉核心交互       | mock 编辑器或拖拽行为，只验证 mock 回调             |
| 5   | 期望值直接来自实现    | 从系统当前输出反推“正确答案”                        |
| 6   | 用 skip 逃避关键场景  | 多人冲突、讨论切换、Markdown 渲染被永久 skip        |
| 7   | 只测 happy path       | 不测保存失败、冲突提示、降级为双栏的分支            |
| 8   | trivial pass          | 仅断言页面能打开或接口返回 200                      |

### 1.4 防护机制

1. **AC 强制溯源**
   - 每个测试必须引用 `AC-FRXXXX-YY` 或 `AC-NFRXXXX-YY`
   - acceptance 中的 `### AC-N` 按所属 FR / NFR 机械规范化为 canonical AC ID；例如 `FR-0100 / AC-1` = `AC-FR0100-01`
   - CI 校验 acceptance 与 tests 的引用闭合

2. **交互断言禁忌**
   - 不允许只用“元素存在”作为复杂交互的唯一断言
   - 不允许把“点击/拖拽/编辑/切换/折叠”场景退化为纯接口测试

3. **可测试性回退**
   - 若某 AC 需要观测实现层状态，必须先补 `interfaces.md`
   - 不允许通过暴露私有 API 或直接读内存对象来测试

4. **架构决策纪律**
   - `FR-0900 AC-3` 已在 `M-ARCH` 锁定为双栏源码编辑 + 实时预览
   - 测试必须验证双栏是最终可观察行为，而不是继续把“曾经讨论过的单窗方案”当成分支验收路径

### 1.5 测试分工

- **单元/集成测试**：实现者与功能代码同步提交
- **浏览器 E2E**：覆盖核心用户流程与多人协作风险
- **评审关注点**：UI 交互契约、Markdown 保真、`inline-discussion` 语义保真、多人冲突提示

## 2. 测试环境

### 2.1 目录组织（推荐）

```text
tests/
├── unit/                # 与具体解析、存储、路由有关的快速测试
├── integration/         # 文档保存、绑定持久化、接口闭合
├── e2e/                 # 浏览器端关键场景
├── fixtures/            # 样例 spec/wiki/acceptance/test-plan 文档
└── assets/              # Markdown 样例、截图基线（如项目采用）
```

### 2.2 命名约定

- 文件：`test_<feature>__<scenario>.*`
- 测试：`test_ac_<fr_id>_<scenario>`
- E2E 场景名应直接对应 Story / Scenario / Acceptance 语义，而非框架术语

### 2.3 执行

- **执行顺序**：unit → integration → e2e
- **默认 CI**：unit + integration + 关键 e2e
- **扩展 CI / nightly**：多人协作冲突、复杂 Markdown 大文档场景
- **浏览器矩阵**：默认至少 1 个主浏览器；是否扩展更多浏览器由 `M-ARCH` 决定

### 2.4 测试数据

- `fixtures/markdown/`：覆盖代码块、表格、列表、嵌套列表、长文档
- `fixtures/discussion/`：覆盖 `inline-discussion` 的发起、回复、折叠、切换
- `fixtures/specs/`：覆盖含 FR/NFR 卡片、锚点、元信息表的设计文档
- `fixtures/wiki/`：覆盖 wiki 页面列表、空页面、已有页面、冲突场景

## 3. Ground Truth 方法

### 3.1 总体结论

本 spec **大部分场景不需要算法型 ground truth**，因为核心验证对象是：

- UI 可交互行为
- Markdown 渲染保真
- 文件持久化结果
- 多人协作可见性

因此 Ground Truth 的重点不是计算结果，而是**样例文档 + 外部可观测行为**。

### 3.2 适用的独立真值来源

- Markdown 样例文件本身，作为渲染真值来源
- `inline-discussion` 协议样例文件，作为讨论渲染与写回真值来源
- 被保存后的 Markdown 文件文本，作为编辑保真真值来源

### 3.3 隔离原则

- 测试中的期望结果不得由系统当前渲染结果反推
- Markdown/讨论样例必须来自 `fixtures/` 的独立样本
- 若采用截图比对，截图仅作补充，不得替代语义断言

## 6. 外部依赖分层测试

> 本 spec 启用。因为系统至少涉及浏览器、HTTP 服务、文件系统，以及可能的多人并发会话。

### 6.1 三个不可回避的约束

| #   | 约束                                                | 后果                             |
| --- | --------------------------------------------------- | -------------------------------- |
| C1  | 需要真实浏览器交互才能验证拖拽、切换、折叠          | 不能只靠纯后端测试替代           |
| C2  | 需要真实文件持久化来验证 Markdown / discussion 保真 | 不能只测内存状态                 |
| C3  | 多人协作需要至少两个会话视角                        | 单会话 happy path 不足以覆盖风险 |

### 6.2 立场：可控化 vs mock

- **可控化**：浏览器会话、测试项目目录、样例 Markdown 文件、测试用服务实例
- **不可 mock**：编辑器核心交互、discussion 渲染、拖拽绑定语义、最终持久化结果

### 6.3 三层测试金字塔

| 层  | 名称       | 覆盖                                      | 默认运行                  |
| --- | ---------- | ----------------------------------------- | ------------------------- |
| L1  | 语义单测   | Markdown 解析、讨论协议写回、绑定状态读写 | ✅                         |
| L2  | 集成测试   | 服务启动、文件持久化、接口与页面状态闭合  | ✅                         |
| L3  | 浏览器 E2E | 拖拽、编辑、切换、折叠、多人冲突与可见性  | ✅ 核心场景 / 其余 nightly |

### 6.4 责任契约

| 组件           | 职责（对外）                               | 边界                   |
| -------------- | ------------------------------------------ | ---------------------- |
| 测试服务实例   | 启动 web server 并提供测试数据目录         | 不改写业务逻辑         |
| 浏览器自动化   | 执行拖拽、编辑、切换、折叠、保存等用户动作 | 不直接操纵内部状态     |
| fixture 文档集 | 提供 Markdown / discussion / spec 样例     | 不依赖运行时生成       |
| 多会话编排器   | 同时驱动至少两个浏览器会话                 | 不替代真实冲突处理逻辑 |

### 6.5 与 interfaces.md 的闭合要求

后续 `interfaces.md` 至少要提供这些可断言出口：

- web server 启动与健康检查
- 模型绑定的读取/保存出口
- 文档加载/保存出口
- discussion 写回后的文件结果
- 多人协作中的更新时间 / 修改人 / 冲突提示出口

- AC-1：L2/L3，三类文档都能在线阅读

- AC-1：L2/L3，修改后存在更新时间或修改人提示
- AC-2：L3，多会话连续修改时存在冲突/覆盖/重载提示

- AC-1：L2/L3，代码块渲染与缩进保真
- AC-2：L2/L3，表格与列表层级保真
- AC-3：L3，验证左侧源码编辑 + 右侧实时预览的双栏工作台行为

### FR-1000 `inline-discussion` 的正文/讨论聚焦与折叠
- AC-1：L3，默认讨论弱化、正文为主
- AC-2：L3，正文/讨论切换后主次对换
- AC-3：L3，讨论可折叠且可重新展开

### NFR-0100 Markdown 兼容与工作流保真
- AC-1：L2，保存后的文件继续被 CLI / 协议工具读取

### NFR-0200 交互稳定
- AC-1：L3，保存、切换、拖拽后无整页重载和明显闪烁

### NFR-0300 面向团队内使用的可部署性
- AC-1：L2/L3，按文档可启动真实服务并完成核心操作
