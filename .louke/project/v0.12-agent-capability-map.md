# v0.12 Agent Capability Decomposition

- **状态**：Proposal
- **创建日期**：2026-07-13
- **适用版本**：v0.12
- **上位目标**：`.louke/project/v0.12-programmatic-workflows.md`

## 1. 目的

本文件不是新的 workflow spec，而是 v0.12 的迁移清单。它逐个记录当前 Agent 的职责如何拆成：

1. 必须迁入程序的确定性职责。
2. 应保留给 Agent 的语义职责。
3. 需要人类授权的职责。
4. 迁移后 Agent 允许使用的工具及禁止的副作用。

任何 Agent 只有在这张表中仍存在语义核心时才继续作为 Agent；没有语义核心的 Agent 在迁移完成后删除。

## 2. 统一边界

### 2.1 程序拥有

- WorkflowRun、current step、revision、retry、timeout 和合法 transition。
- foundation、结构检查、lint/test/coverage、hash、schema validation。
- OpenCode server/session/worktree 的创建、恢复、关闭和记录。
- GitHub Issue/Project、branch、commit、push、tag、发布等权威副作用。
- Agent context manifest、输出 schema 验证、事件与 evidence 持久化。
- human gate challenge 和认证 principal。

### 2.2 Agent 拥有

- 理解自然语言、识别歧义和提出澄清问题。
- 生成 spec、test、architecture、code、review、threat model、knowledge summary 等语义产物。
- 在候选集合中提出选择并解释理由。
- 在受控 worktree 内使用 read/search/edit 和非权威 test 命令获得反馈。

### 2.3 人类拥有

- 合同锁定、waiver、风险接受、发布授权及程序无法可靠决定的产品选择。

## 4. 统一 Semantic Task 合同

程序调用 Agent 时生成不可变请求：

```json
{
  "task_id": "task_...",
  "task_type": "spec_draft|architecture_design|implementation|review|decision|distillation",
  "contract_version": 1,
  "run_id": "run_...",
  "step_id": "step_...",
  "attempt": 1,
  "workspace": {
    "worktree": "/controlled/path",
    "base_commit": "...",
    "branch": "...",
    "dirty_digest": "sha256:..."
  },
  "inputs": [],
  "facts": {},
  "allowed_choices": [],
  "allowed_tools": ["read", "search", "edit", "test"],
  "allowed_write_paths": [],
  "forbidden_effects": ["workflow_state", "approve_gate", "commit", "push", "publish"],
  "output_schema": "semantic-result/v1"
}
```

Agent 返回：

```json
{
  "task_id": "task_...",
  "outcome": "completed|needs_input|unable|rejected",
  "summary": "...",
  "artifacts": [],
  "patch_digest": "sha256:...",
  "decision": null,
  "findings": [],
  "questions": [],
  "confidence": null,
  "evidence": []
}
```

程序必须验证 task id、schema、worktree diff、write allowlist 和禁止副作用，再决定是否接受结果。Agent 输出中的 `next_step`、`approved_by`、`workflow_status`、`commit_sha` 等字段没有权威性。

## 5. 工具分级

| 工具类别 | Agent | 程序 |
|---|---|---|
| read/search | 在 context scope 内允许 | 记录访问范围和输入 manifest |
| edit/生成 patch | 在 allowlist worktree 内允许 | 验证最终 diff、应用/保留产物 |
| 局部 test/lint | 允许用于反馈，不具备 gate 权威 | 使用干净环境权威重跑并记录结果 |
| workflow state/gate | 禁止 | 唯一执行者 |
| commit/push/Issue/tag/publish | 禁止 | 验证后执行 |
| credential/auth principal | 不向 Agent 暴露 | 宿主最小权限管理 |

## 6. 单个 Agent 的迁移步骤

每个角色按同一顺序迁移：

1. 列举当前 prompt 中的每个动作及实际调用的 CLI/API。
2. 为确定性动作建立 program handler 和 contract test。
3. 定义剩余语义任务的 request/result schema。
4. 收紧 Agent tools、write allowlist 和 forbidden effects。
5. 在 shadow mode 同时运行旧路径和新路径，只比较结果，不让旧 Agent 推进状态。
6. 新路径通过 acceptance 后关闭旧程序化职责。
7. 若语义核心为空，删除 Agent；否则保留精简后的 prompt。

## 7. 首批迁移

### 7.1 v0.12-001

- `foundation.ensure` 接管 Scout 的确定性创建/检查。
- Warden F1–F11 合并为 foundation 的程序复验。
- Warden 的 story reasonableness 移到后续 Sage/Lex semantic review。
- `human_gate` 接管 M-LOCK；Maestro 不再持有批准或状态写权限。

### 7.2 v0.12-003

- `quality.gate`、`regression.check` 接管 Keeper，删除 Keeper Agent。
- 抽离 Lex/Sage/Archer 的结构检查、Issue/commit 等副作用。
- 抽离 Devon/Shield 的 worktree、session、权威测试和提交动作。
- 抽离 Prism/Judge/Librarian 的扫描、输入收集、结果存储与发布动作。

## 8. 迁移完成条件

单个 Agent 必须同时满足以下条件才能宣布完成拆分：

1. Prompt 中不存在 workflow advance、gate approval、commit/push/publish 等权威动作。
2. 所有确定性检查都有 program handler 和离线 contract test。
3. 每次调用都有持久化 context manifest 和结构化 result。
4. Agent 越权写路径、返回非法字段或调用禁止工具时，程序拒绝结果且状态不变。
5. 程序在干净环境重跑权威验证，不能信任 Agent 自报成功。
6. 旧 Agent 路径不再被新 WorkflowRun 调用。
7. 若 Agent 无剩余语义职责，其 prompt、模型绑定和调度入口被删除。
