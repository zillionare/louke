# v0.12 程序化工作流 — 交付地图

- **状态**：Proposal
- **创建日期**：2026-07-13
- **目标版本**：v0.12

## 1. 唯一目标

把 Louke 从“Maestro 驱动工作流”改造成“程序驱动工作流”：

- Louke 程序是工作流状态和合法转移的唯一权威。
- Maestro 只在程序声明的决策点提供建议，不能直接推进步骤。
- Devon 等执行 Agent 只完成被分派的任务，不能决定下一阶段。
- 人类门禁由程序持久化并强制执行，不能由 Agent、命令参数或伪造身份绕过。
- 确定性能力由普通程序步骤实现，不再为纯工具调用保留 Agent。

这五个交付切片共同服务于上述一个目标，不是五套互不相关的功能。

## 2. 程序—Agent 分离原则

v0.12 采用一条贯穿所有切片的架构原则：

> 完全抽离 Agent 的确定性职责、流程控制权和可程序化副作用；Agent 只承担需要理解、权衡、审查或创造的语义任务。

判断一项职责归属时使用以下规则：

| 特征 | 归属 |
|---|---|
| 相同输入应产生相同结果，规则可以枚举 | 程序 |
| 存在多个合理答案，需要解释、权衡或创造 | Agent |
| 涉及授权、风险接受或不可逆决定 | 人类 |

“Agent 只做语义任务”不等于禁止 Agent 使用 read/search/edit/test。Devon 等 Agent 可以在程序创建的隔离 worktree 和明确 allowlist 内使用探索、创作工具；但不能写 WorkflowRun、批准 gate、创建权威 commit/push、决定下一步骤或伪造程序验证结果。

每个 Agent 节点遵循同一握手：

```text
程序准备事实和边界
    -> Agent 产生语义结果/补丁/审查意见
    -> 程序验证 schema、权限、diff 和确定性门禁
    -> 必要时等待人类
    -> 程序执行副作用并改变 workflow state
```

逐 Agent 的现有职责、待抽离能力和保留语义核心记录在 `.louke/project/v0.12-agent-capability-map.md`。

## 3. 目标结构

```text
Web / CLI
    |
    v
Louke Workflow Runtime                 唯一流程控制者
    |-- program step                   确定性程序
    |-- human gate                     人类授权
    |-- decision step -> Maestro        有限选项中的建议
    `-- agent task -> OpenCode session  Devon/Reviewer 执行
```

## 4. 候选交付切片及依赖

> 本节是在 M-SPEC 前形成的探索性分期，不是锁定需求或实施命令。正式模块和交付切片必须在完整 story/spec/acceptance 经用户与 Lex 批准后，由 M-TESTPLAN/M-ARCH 重新推导；若正式设计与本节冲突，以锁定合同和正式设计为准。

### v0.12-001 — Programmatic Workflow Runtime

建立最小但真实的工作流内核：版本化定义、持久化运行、合法转移、程序步骤、人类门禁、事件记录和崩溃恢复。

首批迁移：

- 把 M-FOUNDATION 拆成幂等的 `foundation.ensure` 程序前置条件。
- 把 M-LOCK 变成不可绕过的 `human_gate`。
- Maestro 不再拥有新运行实例的状态写权限。

完成出口：即使调用者直接请求后续步骤，未批准的 M-LOCK 也无法越过；重启 Louke 后仍停留在同一门禁。

### v0.12-002 — OpenCode Runtime and Step Context

接入真实 OpenCode server/session，明确 server、workspace、session、client 四种生命周期；为每个工作流步骤构建可重放的 context manifest、隔离 worktree 和工具权限边界。

完成出口：Web 断开不终止 server/session；可重新 attach；Devon 的一次任务能证明它获得了哪组输入、基于哪个 commit 工作。

### v0.12-003 — Semantic Agent Contracts and Capability Extraction

定义统一的 `SemanticTaskRequest/Result`、Agent 工具权限和产物交接协议；按能力地图把各 Agent 内的结构检查、环境准备、commit/push、结果持久化等确定性部分迁入程序。

首批完整迁移 Keeper；继续拆分 Sage、Lex、Archer、Devon、Shield、Prism、Judge 和 Librarian 的程序部分，同时保留其语义核心。

完成出口：每个 Agent 调用都有受版本约束的输入 manifest、允许工具、允许写路径和输出 schema；Agent 不能写 Runtime state 或执行权威发布副作用；纯工具 Agent Keeper 不再被启动。

### v0.12-004 — Dynamic Workflow Catalog and Routing

增加 `agent_task` 和 `decision` 节点。Maestro 只能在 definition 声明的候选项中提出选择；程序校验输出后决定转移。提供 feature、bugfix、spec-change 三个可选择工作流，并把新的程序/Agent 职责边界用于真实流程。

完成出口：同一入口可以选择多个固定工作流；工作流内部可以在预先声明的边上动态分支，但 Agent 不能发明、跳过或直接执行流程转移。

### v0.12-005 — Web Control Plane and Legacy Migration

Web 展示 WorkflowRun、当前步骤、等待原因、Agent 会话和人类门禁；把现有 Maestro pipeline 迁移为工作流定义，并关闭 Maestro 直接 advance 新运行实例的路径。

完成出口：日常 feature 开发可由 Web 启动、暂停、恢复和审计；旧命令只用于读取历史或显式兼容模式。

依赖关系：

```text
001 workflow authority
   -> 002 OpenCode execution
       -> 003 semantic-agent boundary
           -> 004 workflow routing/catalog
               -> 005 UI + full migration
```

## 5. 文档放置规则

每个可独立验收的切片使用一个 spec 目录：

```text
.louke/project/specs/<spec-id>/
    story.md          原始问题、目标和边界
    spec.md           FR/NFR；描述必须实现什么
    acceptance.md     外部可观察的通过条件
    test-plan.md      如何证明 acceptance
    architecture.md   模块、状态和数据流设计
    interfaces.md     API、数据结构、错误和状态机契约
```

本文件只负责跨 spec 的目标、顺序和边界，不作为某个实现的验收合同。运行状态、OpenCode 会话和临时产物不得写入 spec 目录。

## 6. 当前 Louke 流程的使用方式

在 v0.12-001 实现自身完成前，现有 Louke 只能作为辅助工具使用：

- 可用：Sage/Lex 的需求检查思想、Archer 的 test-plan/architecture 结构检查、Prism 的语义 review、现有 stage-result 哈希机制。
- 暂不用：`Scout foundation`、`Maestro advance`、`record-lock --confirm`、自动 commit/push/create-issues。
- 原因：当前流程仍由调用者指定 stage，尚不能证明 persisted current state 与请求转移一致；M-LOCK 的 `--confirm` 也不是经过身份验证的持久化人类门禁。

草案评审通过后，再由人类明确决定何时把 `project.toml` 的 active spec 从 v0.11-002 切换到 v0.12-001。

## 7. v0.12 总体验收定义

以下条件同时满足时，才可称为“程序化工作流”完成：

1. 任一 WorkflowRun 的当前状态只能由 Runtime 按定义改变。
2. Agent 无法直接写运行状态或选择未声明的下一步。
3. M-FOUNDATION 不再需要 Scout/Warden Agent。
4. M-LOCK 未获得绑定当前合同版本的人类批准时，任何路径都不能进入开发。
5. Louke 重启、Web detach 和 OpenCode client detach 不丢失运行状态。
6. 每个 Agent 步骤都有可重放的输入 manifest 和结构化输出。
7. feature、bugfix、spec-change 至少三条工作流可被选择并独立演进。
8. 每个保留 Agent 的所有确定性职责均已迁入程序；Agent 只接收语义任务合同，并且不能控制工作流状态、人类授权或权威发布副作用。
