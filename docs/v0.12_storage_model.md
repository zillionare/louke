# v0.12 Persistence Model - Two Stores

The v0.12 product uses **two independent persistence layers** that serve
different concerns. They are **not unified** in v0.12; unification is a
v0.13 follow-up (see §Deferred below).

## The two stores

| Layer | Module | Backend | Owns |
|---|---|---|---|
| **Web static store** | `louke/web/store.py` | JSON files (`.louke/project.toml`, `.louke/users.json`, wiki, docs) | Local human principal credentials, wiki/doc contents, project metadata |
| **Runtime store** | `louke/runtime/store.py` | SQLite (`.louke/runtime.sqlite3`) | WorkflowRun, StepAttempt, Gate, Event, project content records |

## Why two stores

These layers evolve independently:

- The **web store** is read by the Web UI on every page render; v0.11's auth
  and wiki routes have used JSON for years. Migrating to SQLite would force
  a rewrite of every wiki render and every auth check.
- The **runtime store** must support ACID transactions, revision CAS, and
  recovery after crash - properties JSON cannot provide. SQLite is the
  correct fit.

Forcing them into one store would mean picking the lower common
denominator (JSON, losing ACID) or migrating all web consumers to SQLite
(massive churn with no user-visible benefit).

## How they interact

- `lk serve` writes `.louke/project/project.toml` via the web store (v0.11
  contract). The runtime store reads `spec_id` from the same file to bind
  a WorkflowRun to a spec.
- A v0.12 web page that lists projects calls `GET /api/projects/*` (B1 sub-app),
  which uses the runtime store. Pages that show wiki/auth call web store APIs
  directly (v0.11 contract, unchanged).
- The OpenCode `ManagedInstanceState` (B4) lives in its own JSON file at
  `<workspace>/.louke/opencode/instances.json` - a third concern, separate
  from both web and runtime stores.

There is no data duplication. Each fact lives in exactly one place.

## Deferred (v0.13)

Per the QoderWork gap-analysis §3 (Gap-3):

> 持久化层双轨 (架构风险)
> 两套独立存储系统并存但互不通信

This is a known architectural concern. Unification proposals include:

1. **Adopt runtime SQLite for everything**: requires rewriting wiki/auth
   rendering paths; estimated churn is large.
2. **Adopt web JSON for runtime state**: loses ACID, revision CAS, and
   concurrent write safety - directly violates AC-NFR0001-01 and
   AC-NFR0101-01.
3. **Single SQLite file with separate schemas**: technically possible but
   gains nothing - the `WorkflowRunStore` and `ProjectStore` already have
   different access patterns; co-locating them only saves a file.

v0.13 will revisit (1) once the v0.12 product stabilises and the wiki/auth
churn is more affordable.

## What v0.12 actually shipped

| Concern | Store | Path | Notes |
|---|---|---|---|
| First user / credentials | web | `.louke/users.json` (HMAC-hashed) | v0.11 pattern; NFR0401-02 |
| Wiki / docs | web | `.louke/wiki/*.md` | v0.11 pattern; unchanged |
| Project metadata | web | `.louke/project/project.toml` | written by `lk serve` (B2) |
| Projects (active / history / backlog) | runtime | SQLite `projects` table | new in v0.12 |
| WorkflowRun / StepAttempt / Gate / Event | runtime | SQLite tables | new in v0.12 |
| OpenCode instance states | opencode | `.louke/opencode/instances.json` | new in v0.12 |
| OpenCode recovery scan | opencode | same file (overwritten in-place) | new in v0.12 |

## Acceptance

| AC | Where satisfied |
|---|---|
| AC-NFR0001-01 (atomicity / crash safety) | runtime store (SQLite transactions + recovery) |
| AC-NFR0101-01 (concurrency / CAS) | runtime store (`expected_revision` CAS in `GateService.submit_decision`, `WorkflowOrchestrator.apply_command`) |
| AC-FR1801-01 (init wizard idempotent) | both: web store for `project.toml`, runtime store for projects |
| AC-FR2301-01 (legacy preview / preserve) | legacy: web store (preserve); new v0.12 records: runtime store |
| AC-FR2401-01 (concurrent projects isolated) | runtime store: each workspace has its own SQLite file at `<workspace>/.louke/runtime.sqlite3` |
| AC-NFR0401-02 (credentials not plaintext) | web store: HMAC-hashed via `CredentialStore` (NFR0401) |
