## 1. Executive summary

`v0.12.1` is a stabilization patch over the `v0.12.0` tag. It does **not** add
product features; it fixes five release-blocking defects that made `v0.12.0`
uninstallable as a wheel, shadowed a production API route, ran no v0.12 test
gate in CI, conflated test layers, and lacked an installed-wheel / browser
smoke. The shipped `v0.12.1` wheel now contains the full v0.12 surface
(`louke.runtime`, `louke.opencode`, `louke.web.api`, `louke.web.pages`,
`louke.cli_v12`), `lk --version` reports `0.12.1` from a clean install, the
production `/api/runtime/bindings/*` endpoints are reachable through the
top-level `create_app()`, CI runs unit + integration + ground_truth with a
95.81% coverage gate (above the 95% floor), one Chromium product journey
passes, and the M-E2E/DoD evidence now distinguishes `passed` / `skipped` /
`waived` / `not-run` honestly. The `v0.12.0` tag is deleted (see §7) because
it shipped an uninstallable wheel; users who installed `v0.12.0` must
reinstall `v0.12.1`.

## 5. Pass / skip / waive / not-run distinction

Per gap-analysis §3 P1-3 and §8, the four states are strictly distinguished.
No `waived` or `not-run` gate is reported as `passed`.

| State | Meaning | Instances in v0.12.1 |
|-------|---------|-----------------------|
| **passed** | Test ran and asserted product behavior; gate executed and met its threshold. | 443 unit+integration+ground_truth; 180 non-browser E2E; 1 Chromium journey; coverage 95.81% ≥ 95%. |
| **skipped** | Test was collected but did not run due to an environment condition (no browser, no real OpenCode server, missing wheel-build fixture). Reported as skipped, NOT as pass. | 2 in unit+integration+ground_truth (`test_package_smoke` markers); 7 in non-browser E2E (browser/real-opencode env); Chromium journey skips cleanly when `chromium-1228` is absent. |
| **waived** | A gate's verdict was `fail` in history, and a human accepted a documented risk to allow continuation. The original `fail` verdict is NOT rewritten to `pass`; only the control decision changes. | The 2 baseline gate warnings from `v0.12.0` (M-E2E verdict + DoD wording mismatch) are the waived items. They are carried forward as history, not re-litigated; `v0.12.1` produces independent fresh evidence (this audit + Batches 1–5) rather than editing the historical JSON. |
| **not-run** | A gate exists in the plan but was not executed in this release context. | Multi-browser matrix (out of v0.12 scope per spec); real OpenCode L3 in CI (requires live `opencode serve`, run locally per runbook `docs/v0.12_l3_opencode_smoke_runbook.md`, not in default CI). |

The historical `v0.12.0` waivers are:
1. `M-E2E/gate-result.json` verdict = `fail` (AC trace coverage gap), waived by

as an audit trail.

endpoints through the top-level `louke.web.app.create_app()`:

### v0.12.0 deletion rationale

The `v0.12.0` tag shipped a wheel whose `[tool.setuptools] packages` listed
only `louke`, `louke._tools`, `louke.web` — omitting `louke.runtime`,
`louke.opencode`, `louke.web.api`, `louke.web.pages` (Batch 0 §0.6, Batch 1 §1).
A user who `pip install louke==v0.12.0` would get an import that crashes on
`import louke.runtime` (Batch 0 §0.8). Since the tag points at an uninstallable
artifact and there is no v0.12.0.x hotfix branch, the tag is deleted to
prevent users from pinning to a broken release. The deletion is documented in
the `v0.12.1` tag message (S10) so the history is recoverable from `git reflog`
and the tag message itself. The historical `v0.12.0` fail/waiver JSON artifacts
in `.louke/` are **not** deleted (gap-analysis §6 #3).

runs, in order:
1. Build wheel/sdist
2. Install wheel into clean venv
3. Install test dependencies (`pytest`, `pytest-cov`)

5. Package smoke: import v0.12 subpackages from installed wheel
6. Package smoke: `lk --version` reports `0.12.1`
7. `lk --help` smoke
8. BATS suite (retained)

The coverage gate (`fail_under = 95` in `pyproject.toml [tool.coverage.report]`)
is **not** lowered (gap-analysis §6 #8). Local run: 95.81% (Batch 4 §3).

The secondary `louke-ci.yml` still installs louke from PyPI for backward-
compatibility AC scanning; this is a documented limitation (Batch 4 §6) and a
v0.13 follow-up, not a v0.12.1 release blocker (gap-analysis §3 P0-3 allows
either fix or explicit documentation).

### Honest re-audit at release-time (post-tag fix verification)

The audit lines above were written **before** the v0.12.1 tag was created. A user audit (2026-07-15) found that **CI was actually red at tag time** because:

1. pre-commit pipeline (Batch 4 / S6) had latent failures: trailing whitespace in `batch0_baseline.txt`, missing newline in `louke/web/api/_common.py`, ruff F821 forward-ref in `tests/unit/opencode/test_recovery.py:23`, mypy duplicate-conftest in `tests/integration/conftest.py`. CI log run #29405319569 records the failures. Fix landed as commit `eac37d9` post-tag.
2. ci.yml pytest step ran in system python3 (no `httpx` / `starlette` / `fastapi` installed), so the new tests in `tests/integration/runtime/` collection-failed with `ModuleNotFoundError`. Fix landed as commit `d8cfee2` post-tag.
3. ci.yml bats step similarly lacked deps; ran against system python3 → 380 bats tests poisoned by `ModuleNotFoundError: No module named 'starlette'`. Fix landed as commit `78a65e6` post-tag.

**Honest disclosure**:

- Locally: all 443 unit+integration+ground_truth + 180 non-browser E2E + 1 Chromium journey + 8 BATS verify-acceptance tests pass on `main` after `78a65e6`. 95.81% coverage.
- On GitHub: CI run #29405319569 was red at tag time. CI run #29408532327 (post-tag, after `78a65e6`) was in progress at audit-time and was not waited to completion before this section was written. **The author of this audit cannot claim CI green at tag time without lying.**
- `All gates green` (line below in the previous render of this file) is therefore **misleading and incorrect as of 2026-07-15** and is replaced by this section. A separate, narrower claim — "all Batch gates green locally, and the open-source-style gates from §6 pass on the local checkout" — is true.

Correct status: **`v0.12.1` is releasable as code and evidence; CI status at tag time was red but is fixed post-tag.** The CI re-run after `78a65e6` determines whether `v0.12.1` should keep its tag or be replaced by `v0.12.2`.
